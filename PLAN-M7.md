# Djan-Go-Go -- Milestone 7: Fidelity + polish

> Detailed plan for M7: measured Django fidelity and a real onboarding path. Adds a
> Django-as-oracle differential test harness, `startproject`/`startapp` scaffolding,
> docs, and an example app. Conventions: TDD where applicable, conventional commits
> straight to `main`, `gofmt`/`go vet`/`golangci-lint` (0 issues) per commit. No
> em-dashes (use `--`). Go on PATH: `export PATH="$HOME/.local/go/bin:$HOME/go/bin:$PATH"`.
> Python3 + Django 6.0.3 are available for the oracle.

**Goal:** Differential tests measure fidelity against real Django; `djangogo
startproject` produces a runnable project; docs and an example app exist.

**Exit criteria:** the differential template-fidelity tests pass; `startproject`
generates a project whose Go sources build; `go test ./...`, `go vet`,
`golangci-lint` green.

---

## Work packages

### W1 -- scaffolding (`startproject` / `startapp`)
Package `scaffold` plus two manage commands registered on the Application:
- `scaffold.Project(dir, name string) error`: writes a runnable project skeleton:
  `go.mod` (module `<name>`, `require github.com/oliverhaas/djangogo` + a `replace`
  to the local checkout so it builds without publishing), `main.go` (builds a
  `djangogo.Application` with a demo app + `runserver`), `settings.go` (a
  `conf.Settings` with SecretKey + a sqlite Database), and a `<name>app/` app package
  (models.go with a sample model, admin.go registering it, urls.go).
- `scaffold.App(dir, name string) error`: writes an app package skeleton (models.go,
  admin.go, views.go, urls.go, an empty migrations dir marker).
- `startproject`/`startapp` manage commands wrapping these (in the root `djangogo`
  package), wired into the Application command set.
- Tests: generate into `t.TempDir()`; assert every file exists and PARSES as valid Go
  (`go/parser`); assert the `go.mod` is well-formed; and (exit-criterion proof) run
  `go build ./...` inside the generated project with the replace pointing at the repo
  root, asserting it compiles. (If a full build is too heavy/slow in the test, at
  minimum `go vet`/parse all generated files and build the project in one
  representative test.)

### W2 -- Django-as-oracle differential harness (template fidelity)
Directory `fidelity/`:
- `fidelity/oracle/render.py`: a standalone Django-configured Python script that
  renders a fixed set of canonical template+context cases (defined in a shared
  `cases.json`: name -> {template, context}) using Django's template engine, and
  writes `golden/<name>.txt` (the DTL output) plus echoes the Django version.
- `fidelity/cases.json`: canonical cases chosen to be DTL/pongo2-faithful -- variable
  interpolation, `{% for %}` with `{{ forloop.counter }}`, `{% if %}/{% else %}`,
  filters (`upper`, `lower`, `default`, `length`, `join`), and HTML autoescaping of a
  value with `<>&"`.
- `fidelity/golden/*.txt`: committed golden outputs generated by the script (run it
  now with the available Django 6.0.3).
- `fidelity/render_test.go`: renders each case with the project's pongo2 `templates`
  engine and asserts the output equals the committed golden file. Cases where pongo2
  has a DOCUMENTED cosmetic divergence from DTL are either normalized or listed in a
  `divergences.md` and skipped with a clear `t.Skip` reason (do not hide real
  mismatches -- only skip genuinely-documented cosmetic ones). The Go test reads the
  committed golden files, so it does NOT require Python at test time.
- `.github/workflows`: add a CI step (or extend ci.yml) that regenerates the golden
  files via the Python oracle and fails if they drift (so fidelity is regression-
  guarded). Document how to run the oracle locally.
- `fidelity/README.md`: explains the harness and lists known DTL-vs-pongo2 divergences.

### W3 -- docs + example app + final review
- Expand `README.md`: what Djan-Go-Go is, the milestone status (M1-M7), a quickstart
  (define a model -> migrate -> admin), the package map, and the Django-fidelity
  story. Update `CHANGELOG.md`.
- `examples/blog/`: a small but real example app (a `Post` model, admin registration,
  a list/detail view via templates and generic views, urls) wired into a runnable
  `examples/blog/main.go` (using a replace to the repo). A test that builds it and/or
  exercises its handlers via httptest.
- Update `cmd/djangogo/main.go` if useful (e.g. expose startproject/startapp) and
  ensure `go run ./cmd/djangogo` still works.
- Final M7 review: differential tests pass, scaffolding builds, docs accurate, whole
  repo green (vet/fmt/lint), and a holistic pass over the finished framework.

Scope note: the oracle covers TEMPLATE fidelity (the clearest signal); query-SQL and
migration-SQL differential comparison are noted as future extensions (Django's SQL
differs in quoting/structure, making byte-comparison low-value without a normalizer).
